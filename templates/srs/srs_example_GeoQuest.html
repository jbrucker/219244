 1.0 Introduction

1.1 Purpose of this Document

The purpose of this Software Requirements Specification (SRS) document is to give an overview the requirements for the Adventures in Geometry project.  The SRS will define how our team and the client envision the final product and the characteristics or functionality it must have. We note  interface or implementation requirements as well as product functionality that is desired but not absolutely necessary.

1.2 Scope of the Development Project

The product will be a web-based application to assist elementary school children (specifically, 6th graders) with concepts of geometry.

1.3 Definitions, Acronyms, and Abbreviations

  • BROWSER: A browser is an application program that provides a way to look at and interact with all the information on the World Wide Web. The word "browser" seems to have originated prior to the Web as a generic term for user interfaces that let you browse text files online. By the time the first Web browser with a graphical user interface was invented (Mosaic, in 1992), the term seemed to apply to Web content, too. Technically, a Web browser is a client program that uses the Hypertext Transfer Protocol (Hypertext Transfer Protocol) to make requests of Web servers throughout the Internet on behalf of the browser user. A commercial version of the original browser, Mosaic, is in use. Many of the user interface features in Mosaic, however, went into the first widely-used browser, Netscape Navigator. Microsoft followed with its Microsoft Internet Explorer. Today, these two browsers are highly competitive and the only two browsers that the vast majority of Internet users are aware of. Although the online services, such as America Online, Compuserve, and Prodigy, originally had their own browsers, virtually all now offer the Netscape or Microsoft browser. Lynx is a text-only browser for UNIX shell and VMS users. Another recently offered browser is Opera.
  • COOKIE: A cookie is information that a Web site puts on your hard disk so that it can remember something about you at a later time. (More technically, it is information for future use that is stored by the server on the client side of a client/server communication.) Typically, a cookie records your preferences when using a particular site. Using the Web's Hypertext Transfer Protocol (Hypertext Transfer Protocol), each request for a Web page is independent of all other requests. For this reason, the Web page server has no memory of what pages it has sent to a user previously or anything about your previous visits. A cookie is a mechanism that allows the server to store its own information about a user on the user's own computer. You can view the cookies that have been stored on your hard disk (although the content stored in each cookie may not make much sense to you). The location of the cookies depends on the browser. Internet Explorer stores each cookie as a separate file under a Windows subdirectory. Netscape stores all cookies in a single cookies.txt fle. Opera stores them in a single cookies.dat file.
  • GUI: A GUI (usually pronounced GOO-ee) is a graphical (rather than purely textual) user interface to a computer. As you read this, you are looking at the GUI or graphical user interface of your particular Web browser. The term came into existence because the first interactive user interfaces to computers were not graphical; they were text-and-keyboard oriented and usually consisted of commands you had to remember and computer responses that were infamously brief. The command interface of the Disk Operating System operating system (which you can still get to from your Windows operating system) is an example of the typical user-computer interface before GUIs arrived. An intermediate step in user interfaces between the command line interface and the GUI was the non-graphical menu-based interface, which let you interact by using a mouse rather than by having to type in keyboard commands.
  • HTML: HTML (Hypertext Markup Language) is the set of "markup" symbols or codes inserted in a file intended for display on a World Wide Web browser. The markup tells the Web browser how to display a Web page's words and images for the user. The individual markup codes are referred to as elements (but many people also refer to them as tag).
  • JAVASCRIPT: JavaScript is an interpreted programming or script language from Netscape. It is somewhat similar in capability to Microsoft's Visual Basic, Sun's Tool Command Language, the UNIX-derived Practical Extraction and Reporting Language, and IBM's Restructured Extended Executor. In general, script languages are easier and faster to code in than the more structured and compiler languages such as C and C++. Script languages generally take longer to process than compiled languages, but are very useful for shorter programs.

2.0 General Description

2.1 User Personas and Characteristics

Jane, a 13-year-old female with no computer knowledge
 Jane is a 13-year-old female in the 8th grade.  She has never owned or operated a computer in her life.  Jane is not very bright with geometry and wants to learn it.  She is looking forward to improving her geometry skills through any tool provided to her.  She needs extra help with geometry. She is also willing to learn how to operate a computer.

Bob, an 11-year-old male computer power-user
Bob is an 11-year-old computer guru. He is a hacker wannabe and a very intelligent boy who is very good with geometry.  His parents bought him a computer when Bob was six years old, and he has been operating the computer since then.  Bob spends more than 10 hours per day on the computer. Perhaps as a result, he has developed color-blindness and has developed vision problem with small objects.

Teacher Ami, a sixth-grade teacher who is relatively competent in the use of computers
Ami used to be an art teacher and now she is an elementary and junior high school teacher.  She is very good with drawing and loves to judge people’s artwork.  Ami owns a computer and has the skill to operate a computer without difficulty.  She knows how to install software on a computer.   She has extensive knowledge about the Internet.

2.2 Product Perspective

  • This project is an extension to the GeoQuest of Fall, 98. It is a web-based application.
  • Application will interact with a text reader to assist users who have visual disabilities
  • It consists of a graphical user interface written in Java, JavaScript, HTML, or a combination of these.

  • The product is web accessible and must work with the client's existing hardware. A web browser like Netscape 4.0+ or Internet Explorer 4.0+ will be needed.

  • Requires sound card and appropriate audio equipment.

  • Computers will need Internet access with a connection speed of at least 28.8 for decent download of information like pictures and sound. Our product can run locally if all the source files are on the local machine.

2.3 Overview of Functional Requirements

  • A product main page will present the objective of the game and state requirements for running the game. It will also include a link to start the program.

  • An introductory page will introduce the setting of the game and state the mission of the apprentices. This page will allow users to choose between adventure mode and tutorial mode.

  • The tutorial mode will offer lessons and practice problems similar to those asked in the game mode.

  • During the game mode, the users will be asked a series of geometry questions. If the questions are answered correctly, the user will be allowed to proceed. Once the users answer three questions incorrectly, the game is over.      

  • The questions in the game mode will be different each time a new game is started.

2.4 Overview of Data Requirements

A user sends inputs to the software program by either typing keys or clicking on the mouse. The user will have an option to choose either the tutorial mode or to start the game by clicking on the appropriate icon. The user will navigate the tutorial pages by clicking on the mouse button. The quiz questions can be answered by either entering or selecting numbers.

In the adventure mode, the user will traverse a series of frames that pose questions requiring input from either a click-able image map, a text/numerical value entered from the keyboard, or a series of multiple-choice options in the form of radio buttons.  This input will be recorded and, if correct, the user will be transferred from the current frame to the next frame in the adventure.  If incorrect for the third time, the game is over.

In the knowledge assessment mode, there will be comments for both correct and incorrect answers.  For example, when the user answers a question correctly they might get a brief message on their screen, whereas an incorrect answer might give the user two more chances and and then an option to go to the tutorial mode. The student’s score will also be output to the screen as an indicator of his/her performance.  Sound will be incorporated as part of our enhancement techniques, which should make the game more interesting to the children.

2.5 General Constraints, Assumptions, Dependencies, Guidelines

  • The product is available at no charge and is intended for educational purposes.

  • The product will run on any computer with JavaScript-enabled web browsers such as Internet Explorer version 4.0+, or Netscape Navigator version 4.0+.

  • Ideally, the computer will have Internet connection. However, the product should work without Internet connection if copies of the product are stored in local hard drives.

  • File size will need to be considered with incorporation of sound or music.

3.0 Specific Requirements

3.1 External Interface Requirements

  • User Interface: At the start of the game the user will be asked to login his/her name. After logging in the user will be given the option of going into the tutorial mode or starting the game. There will be two buttons: Tutorial Mode and Start Game.

    The game page will have two or three frames (one for picture, one for text directions, and another for menu options). Our user interface is designed to be primarily mouse driven in order to facilitate ease of use. The mouse will be used to allow the user to navigate through out the game and to select answers to questions. There are some questions where the answers have to be typed in.

  • Hardware Components: The game will be able to run if the computer is at least a PowerPC Macintosh or a Pentium-class PC, 32 MB of RAM, and has a color display.
  • Software Components: A web browser, either Internet Explorer 4.0+ or Netscape 4.0+, will be necessary. To hear the sound the user would need to have the proper plug-in for the browser, but this is not a requirement. If the user does not have the plug-in, he/she will not hear the sound but will still be able to play the game. The program has no other software requirements.

3.2.1 Template for describing functional requirements

purpose

a description of the functional requirement and its reason(s)

input

which inputs; in what form/format will inputs arrive; from what sources input will be derived, legal domains of each input element

processing

describes the outcome rather than the implementation; include any validity checks on the data, exact timing of each operation (if needed), how to handle unexpected or abnormal situations

3.2.2 Main Page

purpose

Presents the objective of the game and other requirements and an login option.

input

Input will be mouse driven.

processing

Begins the introduction of the game.

output

Takes the users to the introductory page of the game where options are given to start the game, go to the tutorial mode, or exit the program.

 3.2.3 Game Introductory Page      

purpose

State settings of the game and the mission of the apprentices.

input

Input will be mouse driven.

processing

Begins the game, tutorial mode, or exit the program.

output

Either takes the user to the game page, the tutorial page, or exit the game.

3.2.4 Tutorial Mode

purpose

Offer lessons and practice problems.

input

Input will be mouse driven.

processing

Begins the game after the user is sure he/she wants to move on to the game phase.

output

Three alternatives: takes the user to the game page; goes back to the tutorial mode for more practice; quits the application.

3.2.5 Game Mode

purpose

Solve the geometric puzzle and move forward. The goal is to escape from the castle by answering all the questions.

input

Input will be keyboard mouse driven.

processing

Three incorrect answers would give option to the user either to go into the tutorial mode or exit the game. A correct answer takes the user to the next screen and towards achieving his/her goal.

output

Either an option to start over the game, which will lead to the beginning of the game mode or to exit the game.

3.3 Performance Requirements

  • The user will run the product from the s2s web site hosted by The University of Texas at Austin's Computer Science Department. The number of connections to the system depends on the UT CS department web server configuration and capability. However, the user will be able to run the program locally by installing all the source files on his hard drive or on a CD.
  • GeoQuest is a web-based stand-alone application. Multiple users should be able to run the software simultaneously as long as the number of connections to the web server does not exceed the configuration limit. However, the more the number of users on the system, the less performance every user experiences while using the product.
  • GeoQuest will load up graphics and sounds on every page and will also open up windows using JavaScript on the runtime. The response time for loading every page depends mostly on the Internet connection speed and also data transmission bandwidth. The response time may be longer when used with a slow connection. For this reason, future enhancement of the product should avoid incorporating unnecessary graphics and sound files on the HTML page.
  • Response time of interactive transaction should not exceed 2 seconds per transaction. However, it may vary depending on the client’s system configuration.

3.4 Quality Attributes

  • The availability of the product solely depends on the availability of the web server of UT CS department. Generally speaking, the product should be available 24 hours a day, 365 days a year.
  • GeoQuest should be a very reliable product since it does not require interaction with the local system it is running on. For an example, the software does not open up any external files on the local system. The product is expected to run smoothly without crashing the browser or the system.
  • The product is maintained by good documentation of code and instructions for end users.
